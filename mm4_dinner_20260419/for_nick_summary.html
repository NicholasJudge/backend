MM4 Wireframe

This note is the compact reference for Robert_MM_4 as built on 2026-04-18. It describes the current model stack, the feature flow between layers, the statistical learning contract, and the control-board surfaces needed to diagnose failure modes.

MM4 preserves the Robert outer-loop structure, but retargets that loop to the final live object. The notebook, control board, frozen eval window, publish path, generated live config, and launch or smoke commands are all MM4-scoped and judged against the joint execution transformer used by the real live runner.

Detailed grouped input appendix: for_nick_inputs.md.

1. Model Inventory

This section is the answer to "what is each actual model, what information does it represent, and what does it emit?"

1.1 World model

The world model is a slow cross-asset state model. It is not an MM4 layer. MM4 receives its outputs from the linked signal row and uses them both as compact planner summary fields and as raw m1_* features.

1.4 MM4 joint execution model

This is the actual final live quote-selection model. It is the only model in the stack that directly scores candidate actions.

2. MM4 Input Blocks

block current MM4 trained contract role
market sequence 40 features local microstructure state over time
forecast block 228 features in the published live artifact compact fcst_* planner summary plus raw m1_* payload, dominated by the world-model surface
inventory block 30 features in the published live artifact inv_*, plan_*, quality_*
candidate block 55 features + 5-style id in the published live artifact candidate-specific execution state

2.1 Forecast block

  • fcst_* is the compact planner-facing summary: aggregate edge, confidence, z-score, freshness, source flags, and horizon-0 summary.
  • In the published live artifact, m1_* is not "everything upstream." It is mostly:
    • compact directional summary fields and probabilities;
    • signal-service carry fields such as target inventory and reservation metadata;
    • the full world-model price/range/volume surface and its aligned error terms.
  • The published live artifact does not currently serialize m1_ideal_state_x in its forecast feature list, even though the runtime stack can compute that sidecar field.
  • The newer in-tree training code discusses a leaner 122-feature forecast block. The artifact linked in this note still stores the older 228-feature forecast contract.

2.2 Inventory and planner block

  • inv_* comes from the age-bucket ledger: signed notional by bucket, bucket entry edge, net inventory, gross inventory, and weighted inventory age.
  • plan_* is the analytic planner state: desired inventory, buy/sell budgets, max net/gross inventory, liquidation urgency, reservation offset, alpha half-life, and MM4 micro-blend terms.
  • In the published live artifact, quality_* is still the smaller stream-health block: collector gap, recent fill rows, recent market rows, micro-data disabled or stale, and private-stream health.

2.3 Candidate block

  • Candidate styles are join_touch, inside_improve, target_anchor, band_edge, other.
  • The published live artifact's 55 candidate features mix local quote geometry with macro state: queue ahead/behind, touch distance, candidate-to-mid, candidate-to-microprice, aggressiveness, expected wait, forecast summary, reservation edge, target inventory, current inventory, and inventory gap.
  • The explicit recent same-side execution-memory features live in newer in-tree candidate builders, but they are not part of the published 55-feature candidate contract stored in the linked artifact.

2.4 How upstream model state reaches the final order decision

  • World-model state reaches MM4 twice:
    • as compact planner summary through fcst_world_*;
    • as raw world forecast and error surfaces through m1_world_*.
  • Directional-model state reaches MM4 twice:
    • as compact planner summary through fcst_final_directional_*;
    • as compact linked-signal carry fields such as m1_forecast_*, m1_direction_prob_*, and m1_signal_strength.
  • The published live artifact does not currently expose the full directional aux surface and does not currently serialize the net-lev ideal-point sidecar field m1_ideal_state_x as an MM4 forecast input.
  • Current-versus-target inventory state is explicit through inv_*, plan_*, inventory_target_notional_usd, inventory_notional_usd, inventory_gap_notional_usd, and inventory_ratio.
  • In the published live artifact, recent quote-quality memory is still mostly implicit. The model sees the current book snapshot, planner state, inventory gap, and replay-trained targets, but not the newer explicit trailing execution-memory block.

3. What MM4 Predicts

MM4 is JointMicroTransformerV1 in nv/joint_mods/scorer.py. The current mode is decision_query.

output head group outputs live use
inventory-control heads inventory_residual_frac, inventory_gate, micro_alpha_short_bps, micro_alpha_long_bps fed back into the planner before order construction
candidate-scoring heads fill_logits/prob, adverse_logits/prob, edge_short_ticks, edge_long_ticks, value_score used to rank JR_exec candidates

This is the main MM4 distinction from MM1. MM4 does not only estimate desired inventory. It also estimates execution quality at the candidate level and uses a second planner pass.

4. Statistical Learning Contract

4.1 Data regime

  • Training is offline and replay-based, not policy-gradient RL.
  • The data split is explicit and frozen:
    • train: 2026-04-14 15:05:00.068305969+00:00 to 2026-04-16 15:04:59.606575012+00:00
    • eval: 2026-04-16 15:05:00+00:00 to 2026-04-16 17:05:00+00:00
  • Current dataset manifest:
    • rows: 1,145,175
    • columns: 247
    • eval rows before warmup/gap trimming: 59,452
  • Current causal contract:
    • batch_size=1
    • seq_len=960
    • warm_up_T=480
    • contiguous block batches
    • no contiguous-batch shuffle
    • fresh_random_init_each_run=true
    • resume_existing=false

4.2 Supervision

The loss is a weighted sum of nine heads:

head target type loss current weight
fill binary BCE 0.08
adverse binary BCE 0.35
edge short continuous SmoothL1 0.08
edge long continuous SmoothL1 0.12
value continuous SmoothL1 1.00
alpha short continuous SmoothL1 0.02
alpha long continuous SmoothL1 0.04
inventory residual continuous SmoothL1 0.06
inventory gate continuous SmoothL1 0.06

Target construction:

  • value is the main ranking target. When available, it comes from offline unit replay: passive entry, pathwise inventory marking, stale-inventory requoting, and a stale-inventory penalty.
  • inventory_residual is a bounded transform of alpha_long.
  • inventory_gate is a bounded magnitude transform of alpha_long.

4.3 Why it is only RL-like

The RL-like component is indirect:

  1. replay converts a candidate action and the subsequent path into economic consequences;
  2. those consequences supervise value, fill, adverse, and inventory-control heads jointly;
  3. the learned inventory heads alter the next planner output;
  4. held-out model selection is based on replay economics and behavioral constraints, not only prediction loss.

What is absent:

  • no policy-gradient update;
  • no Bellman backup;
  • no online exploration term;
  • no direct live RL loop in the current MM4 baseline.

5. Live Decision Path

At each live decision time:

  1. build forecast_features = fcst_* + m1_*;
  2. build inventory and quality features from the ledger and streams;
  3. run the baseline planner with zero micro signal;
  4. run MM4 inventory heads with market state + forecast block + baseline planner state;
  5. rerun the planner with the inferred micro signal;
  6. generate JR_exec candidates;
  7. score candidates with MM4 candidate heads;
  8. select the candidate using predicted edge, fill, adverse risk, inventory gate, and a strong near-touch preference;
  9. log the realized or replay consequence for the next training pass.

This is the main information-flow test. If useful information exists upstream but does not change the final candidate ranking, the failure can occur in forecast aggregation, planner transmission, inventory heads, candidate heads, or the final selection rule.

6. Control-Board Blueprint

The present board already contains:

  • training loss over eval steps;
  • training pnl over eval steps;
  • trade volume and leverage over the eval window;
  • pnl path over the eval window;
  • replay reconciliation rows;
  • carry-slice and leverage-response tables;
  • run-step timeline.

For future MM4 work, the control board should be read as seven diagnostic panels:

panel monitor failure it isolates
data integrity dataset dates, row counts, gap blocks, filtered signal rows, live artifact/config existence stale or inconsistent packet
forecast transmission aggregate edge/confidence/freshness, source flags, world-vs-directional disagreement, error magnitudes useful upstream information not reaching MM4
planner transmission baseline vs rerun plan_*, inventory target, reservation offset, urgency, budgets MM4 inventory heads not changing execution intent coherently
inventory-control heads micro_alpha_*, inventory_residual_frac, inventory_gate over time and by side planner feedback loop failure
candidate heads fill, adverse, edge, value by candidate style and touch distance execution ranking failure
inventory/risk state leverage, long/short occupancy, stale-inventory clearance, side balance constraint failure
replay economics spread, fees, adverse selection, carry, baseline, residual, unit reconciliation pnl accounting failure

Three additions would make the board materially stronger:

  • a forecast-to-plan transmission table: upstream edge/confidence vs resulting plan_desired_inventory_notional_usd;
  • a plan-to-action transmission table: rerun planner outputs vs chosen candidate style, touch distance, and realized fill/adverse rates;
  • a contract-integrity table: latest completed eval step, board-synced step, published artifact existence, generated live-config existence, and unit-consistency verdict.

The correct interpretation is:

  • MM4 now has a real joint-execution scaffold;
  • the feature and supervision contracts are concrete;
  • the control board is beginning to isolate failure sources;
  • the present empirical packet is incomplete and statistically weak.
